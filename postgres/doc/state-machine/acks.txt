# ACK State Machine

This document describes all possible state transitions for ACKs in the task tracker system.

## ACK Types

There are two distinct types of ACKs with different behaviors:

1. **External ACKs**: Learned from Forgejo/Github, created by other contributors
   - Always have status = 'external' (immutable marker for external origin)
   - Immutable once created (we don't control external systems)
   - Used for accounting and reviewer threshold calculations
   - No state transitions allowed

2. **Internal ACKs**: Generated by our system for posting
   - Start with status = 'pending' 
   - Transition through pending -> posted/failed based on posting attempts
   - Can be modified by users (message changes reset to 'pending')
   - Subject to retry logic and posting management

## State Dimensions

An ACK has these state dimensions:
1. **status**: pending -> posted/failed (internal ACKs only), or 'external' (external ACKs, immutable)
2. **message**: Can change (internal ACKs only, resets status to 'pending')
3. **reviewer_name**: Immutable identifier
4. **pull_request_id**: Immutable reference

## State Transitions

### Status Transitions (Internal ACKs Only)

- **pending -> posted**: ACK successfully posted to external system
  - Trigger consideration: None needed (application handles posting)
  - Logging: Required with posting details

- **pending -> failed**: ACK posting failed
  - Trigger consideration: None needed (application handles retries)
  - Logging: Required with failure reason

- **posted -> pending**: User modifies ACK message, needs re-posting
  - Trigger consideration: Auto-reset when message changes
  - Logging: Required with old/new message

- **failed -> pending**: Retry posting after failure
  - Trigger consideration: None needed (application/user initiated)
  - Logging: Required

- **failed -> posted**: Retry succeeds
  - Trigger consideration: None needed (application handles posting)
  - Logging: Required

### Message Transitions (Internal ACKs Only)

- **Any message -> New message**: User updates ACK content
  - Trigger consideration: Reset status to 'pending' if currently 'posted'
  - Logging: Required with old/new message content

### Creation Patterns

- **External ACK creation**: Always created with status = 'external'
  - Trigger consideration: Validate status = 'external' on insert, prevent all updates
  - Logging: Required

- **Internal ACK creation**: Always created with status = 'pending'
  - Trigger consideration: Validate status = 'pending' on insert for internal ACKs
  - Logging: Required

## Business Rules

### External ACK Constraints
1. **Status identification**: External ACKs have status = 'external'
2. **External source control**: External ACK content can change when external source (Forgejo/Github) indicates updates
3. **No internal state transitions**: External ACKs don't participate in posting workflow

### Internal ACK Constraints  
1. **Message changes reset posting**: Changing message when status = 'posted' resets to 'pending'
2. **Retry capability**: Failed ACKs can be retried (failed -> pending -> posted)
3. **User modification**: Users can edit internal ACK messages at any time

### Cross-Type Rules
1. **Unique reviewer per PR**: Only one ACK per (pull_request_id, reviewer_name) combination
2. **Threshold counting**: Both external ACKs and internal 'posted' ACKs count toward reviewer thresholds
3. **Tip commit reference**: All ACKs implicitly reference the tip commit of their PR

## Trigger Implementation

### Database Triggers (Implemented)
1. **Reset status on message change**: When internal ACK message changes and status = 'posted', set status = 'pending'

### Application Logic (Not in triggers)
1. **ACK type detection**: Determine if ACK is external vs internal based on creation context
2. **Posting management**: Handle actual posting to external systems
3. **Retry logic**: Manage failed posting attempts and retry scheduling
4. **Logging**: All state changes logged with context about posting attempts/failures

## Invalid State Combinations

### Prevented by Database Triggers (Implemented)
- Message change without status reset to 'pending' (for posted internal ACKs) - **ENFORCED**

### Valid Operational States
- Internal ACK with status = 'pending' and any message
- Internal ACK with status = 'posted' and stable message (changes reset to pending)
- Internal ACK with status = 'failed' and any message
- External ACK with status = 'external' and message controlled by external source

### Application-Level Validation
- Duplicate reviewer names for same PR (prevented by unique constraint)
- ACK references non-tip commit (application ensures ACKs only reference tip)
- Posting attempts on external ACKs (application logic prevents this)

## Schema Implementation

The schema now distinguishes external vs internal ACKs using the status field:

- **External ACKs**: status = 'external' (immutable, no state transitions)
- **Internal ACKs**: status ∈ {'pending', 'posted', 'failed'} (mutable, state transitions allowed)

This approach eliminates the need for additional fields while making the distinction clear and enforceable through triggers.
